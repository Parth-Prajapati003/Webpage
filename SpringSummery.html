Spring Core

Spring Containers:

Spring IoC (Inversion of Control) Container is the core of Spring Framework. It creates the objects, configures and assembles their dependencies, manages their entire life cycle.

There are 2 types of IoC containers:

  • BeanFactory
  • ApplicationContext

The BeanFactory is the most basic version of IoC containers, and the ApplicationContext extends the features of BeanFactory.

because of additional features, developers prefer to use ApplicationContext over BeanFactory.  

For example :
Autowire in BeanFactory : 

 <bean id="welcomeBean" class="com.geeks.beans.WelcomeBean" autowire="byName"/>

Auto wire in Application context : 

@Autowired
private String name;

Dependency Injection:

DI-Related Annotations:

  1. @Autowired
    • @Autowired annotation is applied to the fields, setter methods, and constructors. It injects object dependency implicitly.
    • Field injection
    class Student { 
	@Autowired
	Address address; 
}
    • Constructor Injection
    class Student { 
    Address address; 
  
    @Autowired
    Student(Address address) { 
        this.address = address; 
    } 
}
    • Setter Injection
    class Student { 
    Address address; 
  
    @Autowired
    void setaddress(Address address) { 
        this.address = address; 
    } 
}

  1. @Qualifier Annotation
    • The @Qualifier annotation is used to resolve the autowiring conflict when there are multiple beans of the same type.
    • The @Qualifier annotation can be used on any class annotated with @Component or on methods annotated with @Bean.

    Example :

     @Component(value="car") 
public class Car implements Vehicle { 

	@Override
	public void start() { 
		System.out.println("Car started"); 
	} 

} 

@Component(value="bike") 
public class Bike implements Vehicle { 

	@Override
	public void start() { 
		System.out.println("Bike started"); 
	} 

}

    @Component
public class VehicleService { 

	@Autowired
	@Qualifier("bike") 
	private Vehicle vehicle; 

	public void service() { 
		vehicle.start(); 
	} 
}

    Vehicle is an interface which has two implements so to specify which one to create as a bean we have to use @Quealifier to specify it.

  1. @Primary Annotation
    • It also used to solve same issue as above
    • This indicates that a particular bean should be given preference when multiple beans are candidates to be autowired

BEAN LIFE CYCLE :

  • If we want to execute some code on the bean instantiation and just after closing the spring container, then we can write that code inside the custom init() method and the destroy() method.

For implementing the custome init() and Destroy();

Using annotations :

  • we need to annotate the custom init() method with @PostConstruct and destroy() method with @PreDestroy.
  • When we annotate a Spring Bean method with PreDestroy annotation, it gets called when the bean instance is getting removed from the context.
  • If your spring bean scope is “prototype” then it’s not completely managed by the spring container and the PreDestroy method won’t get called. 

@PostConstruct
    public void init() throws Exception
    {
      // code
    }

    @PreDestroy
    public void destroy() throws Exception
    {
        //Code
    }

Singleton and Prototype Bean Scopes in Java Spring

Singleton Scope:

If the scope is a singleton, then only one instance of that bean will be instantiated per Spring IoC container and the same instance will be shared for each request

Normally each bean is created as singleton scope.

Prototype Scope:

If the scope is declared prototype, then spring IOC container will create a new instance of that bean every time a request is made for that specific bean.

@Scope("prototype")
@Repository
public class Myclass {
	   //code
}

What is Circular Dependency?

  • a circular dependency is a relation between two or more modules that either directly or indirectly depend on each other to function properly

Ex : Bean A → Bean B → Bean A

  • If we try to run it we will get this exception: 

    BeanCurrentlyInCreationException : Error creating bean with name ‘beanA’: Requested bean is currently in creation: Is there an unresolvable circular reference?

Solution:

  1. @Lazy method :
    • By default, Spring creates all singleton beans eagerly at the startup/bootstrapping of the application context.

    public class A {
    private B b;

    @Autowired
    public A(@Lazy B b) {
        this.b = b;
    }
}

    • when Spring creates the application context, it will create a proxy for the B bean instead of creating the actual bean instance immediately. When the B bean is needed (e.g., when injecting it into A), Spring will initialize the B bean and inject it into A. This breaks the circular dependency because A does not need to be initialized when creating B.

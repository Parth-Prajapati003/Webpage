Que 1 :

What is the process of Spring initialisation ? Complete step by step with demonstration thru logs.

Spring Initialization :

  1. Application Startup: Spring Boot initializes the application context when the application starts.
  1. Profile Check : It checks for the any specific Profile. If not specified it will run as a “default” profile.
  1. Component Scanning: Spring scans the classpath for components, such as beans annotated with @Component, @Service, @Repository, or @Controller.
  1. Bean Registration and Initialization: Spring registers these components as beans in the application context. Also Spring initializes the beans, which involves instantiating them and configuring their properties.
  1. Dependency Injection: Spring injects dependencies into the beans.
  1. Servelet Starts : It starts the server. In our case it starts Tomcat server.

Registered BeanFactoryPostProcessors are called, which allows for modification of bean definitions before they are instantiated. BeanFactoryPostProcessors are part of the Spring IoC container's bean creation process. They work on the bean factory itself, allowing for the customization of bean definitions before the container instantiates any beans. How It Works:
  • The ApplicationContext identifies all beans that implement the BeanFactoryPostProcessor interface during startup.
  • It then invokes the postProcessBeanFactory method on each one.
  • This method can modify the bean definitions, add new bean definitions, or even remove bean definitions.
  • This step occurs before any bean instances are created.

7. BeanPostProcessor Execution

Spring processes the beans with registered BeanPostProcessors, which can modify the bean instances after they have been instantiated but before they are ready for use. BeanPostProcessors come into play after the BeanFactoryPostProcessors and work on the bean instances themselves, rather than the definitions. They are used to perform some actions on the bean instance before and after it is initialized (i.e., before and after its initialization callbacks such as @PostConstruct methods or InitializingBean's afterPropertiesSet). How It Works:
  • The ApplicationContext identifies all beans that implement the BeanPostProcessor interface during startup.
  • It then registers these for later callbacks, which are applied to every bean instance.
  • The postProcessBeforeInitialization method is called before a bean's initialization callbacks are invoked.
  • The postProcessAfterInitialization method is called after the bean's initialization callbacks have been invoked.

What is the difference between Auto-wiring a Service or getting it from ApplicationContext using SpringLocatorFactory?

  1. Auto-wiring a Service:
    • Auto-wiring is a mechanism provided by Spring for automatic dependency injection.
    • When you auto-wire a service, Spring automatically injects an instance of the service into the dependent component (such as a controller, another service, or a bean) based on the specified injection type (e.g., constructor injection, setter injection, or field injection) and the matching bean definition in the application context.
    • Auto-wiring is convenient and commonly used, as it reduces the need for explicit configuration and manual instantiation of dependencies.

  1. Getting it from ApplicationContext using SpringLocatorFactory:

    • Using SpringLocatorFactory, We can typically obtain a reference to the ApplicationContext and then retrieve the desired bean by its name or type.
    • This approach gives more control over the bean retrieval process.
    • However, it may introduce tighter coupling between components and may not follow the recommended dependency injection principles advocated by Spring.

Que : 4

Diff between Auto-wiring using a constructor or separately as a variable ?

Constructor Injection

  • In constructor injection, dependencies are injected through the constructor of a class.
  • Constructor injection is considered a best practice in Spring because it ensures that all required dependencies are provided when an instance of the class is created. This makes the dependencies explicit and immutable once set.
  • Constructor injection promotes better testability, as dependencies can be easily mocked or stubbed during unit testing.
  • Constructor injection also helps in enforcing immutability and promoting thread safety.

Field (Variable) Injection

  • In field injection, dependencies are injected directly into class fields using @Autowired annotations.
  • However, field injection can lead to hidden dependencies and can make the code harder to understand, as dependencies are not explicitly declared in the constructor.
  • Field injection can also make unit testing more challenging, as it may require the use of frameworks like Mockito to mock dependencies.

Automatic Configuration in Spring

Question: How to do automatic configuration? (Why? Because it relieves headaches from Dev people).

Answer:

  1. By Autoscanning

    @ComponentScan enables Spring to automatically discover components from an application's classpath and create them as beans in the Spring application context. @Service, @Repository, and @Controller are specializations of @Component for different purposes but are treated the same by Spring.

    Spring Boot introduces the @SpringBootApplication annotation which includes @Configuration, @EnableAutoConfiguration, and @ComponentScan. This triggers automatic component scanning within the application's package and registers them in the Spring's Application Context.

  2. Along with AutoWiring:

    Spring's Autowiring feature automatically injects dependencies between beans.

    Types of Autowiring:

    1. Properties/ Field Injection
    2. Setter Injection
    3. Constructor Injection

    @Autowired on Field/Properties

    Using @Autowired on a field eliminates the need for explicit getters and setters.

    Example:

    @Component("fooFormatter")
public class FooFormatter {
    public String format() {
        return "foo";
    }
}

@Component
public class FooService {
    @Autowired
    private FooFormatter fooFormatter;
}

    Advantages:

    • Quick, convenient, and reduces boilerplate code.

    Disadvantages:

    • Not suitable for mandatory dependencies as it may lead to uninitialized dependencies.
    • Makes testing harder because dependencies are not explicitly exposed.

    @Autowired on Setters

    Spring can autowire dependencies by calling a setter method.

    Example:

    public class FooService {
    private FooFormatter fooFormatter;

    @Autowired
    public void setFormatter(FooFormatter fooFormatter) {
        this.fooFormatter = fooFormatter;
    }
}

    Advantages:

    • Allows the object to exist without being fully initialized (useful for optional dependencies).

    Disadvantages:

    • Can lead to partially constructed objects if not all setters are called.

    @Autowired on Constructors

    Dependencies can also be injected through constructors.

    Example:

    public class FooService {
    private final FooFormatter fooFormatter;

    @Autowired
    public FooService(FooFormatter fooFormatter) {
        this.fooFormatter = fooFormatter;
    }
}

    Advantages:

    • Ensures a bean is never constructed without its necessary dependencies.
    • Ideal for creating immutable objects.
    • Better for testing as dependencies can be explicitly provided.

    Disadvantages:

    • Can lead to large constructors with many dependencies and potentially cyclic dependencies.

Conclusion

In practice, a combination of constructor and setter injection is often seen, with constructor injection used for mandatory dependencies and setter injection for optional ones. Field injection is less preferred and is being discouraged for new Spring development.

  • Constructor Injection is recommended for mandatory and immutable dependencies and is easier for testing.
  • Setter Injection is suitable for optional or changeable dependencies.
  • Field Injection should be used sparingly and is not recommended for critical or complex applications

How to deal with Cyclic dependency ?

  1. @Lazy method :
    • By default, Spring creates all singleton beans eagerly at the startup/bootstrapping of the application context.

    public class A {
    private B b;

    @Autowired
    public A(@Lazy B b) {
        this.b = b;
    }
}

    • when Spring creates the application context, it will create a proxy for the B bean instead of creating the actual bean instance immediately. When the B bean is needed (e.g., when injecting it into A), Spring will initialize the B bean and inject it into A. This breaks the circular dependency because A does not need to be initialized when creating B.

  1. Using @PostConstruct
    • Another way to break the cycle is by injecting a dependency using @Autowired on one of the beans and then using a method annotated with @PostConstruct to set the other dependency.

    Example :

    public class BeanB {

    BeanB() {
        System.out.println("BeanB constructor called !!!");
    }

    private BeanA beanA;
    
    @PostConstruct
    public void init(){
        beanA.setBeanB(this);
    }

    @Autowired
    public void setBeanA(BeanA beanA) {
        System.out.println("Setting property beanA of BeanB instance");
        this.beanA = beanA;
    }
}

And don’t inject dependency using @Autowired in other class. Keep it plain as below.

@Component
public class BeanA {

    BeanA() {
        System.out.println("BeanA constructor called !!!");
    }

    private BeanB beanB;

    public void setBeanB(BeanB beanB) {
        System.out.println("Setting property beanB of BeanA instance");
        this.beanB = beanB;
    }
}

4. Implement ApplicationContextAware and InitializingBean

Implement ApplicationContextAware and InitializingBean to allow a bean to interact directly with the Spring container and perform initialization after all beans are set.

Advantages:

  • Gives you full control over bean initialization.
  • Useful for complex dependency scenarios.

Disadvantages:

  • Increases coupling with Spring-specific interfaces.
  • More complex and can lead to boilerplate code.

Example:

        
@Component
public class A implements ApplicationContextAware, InitializingBean {
    private B b;
    private ApplicationContext context;
 
    @Override
    public void setApplicationContext(ApplicationContext applicationContext) throws BeansException {
        this.context = applicationContext;
    }
 
    @Override
    public void afterPropertiesSet() throws Exception {
        this.b = context.getBean(B.class);
    }
}

@Component
public class B {
    // Similar setup for B if needed
}

Conclusion

Question: So which method out of the above 4 methods to choose?

Answer: When choosing a method to resolve cyclic dependencies, consider the following:

  • Simplicity vs. Complexity: Prefer simple solutions (@Lazy, Setter Injection) for straightforward scenarios. Use more complex approaches (ApplicationContextAware, InitializingBean) for intricate setups.
  • Initialization Timing: If you need to ensure all dependencies are fully initialized before use, @PostConstruct and InitializingBean are more suitable.
  • Coupling: ApplicationContextAware and InitializingBean increase coupling with Spring, which might affect testability and portability.

In general, aim to design your components to avoid cyclic dependencies altogether. This might involve redesigning your class structure or introducing interface-based design to decouple components more effectively.

Is getting a cyclic dependency at first place is a precedent for wrong design or there can be a valid use case for this ?

  • When we have a circular dependency, it’s likely we have a design problem and that the responsibilities are not well separated. It is not following the “Single Responsibility Principle”.
  • We should try to redesign the components properly so that their hierarchy is well designed and there is no need for circular dependencies.
  • However, there are many possible reasons we may not be able to do a redesign, such as legacy code, code that has already been tested and cannot be modified, not enough time or resources for a complete redesign, etc.
  • So in nutshell, Having a circular dependencies shows it is a bad design.

what is MOCKITO @Befor and @After ?

  1. @Before:
    • @Before is an annotation used to mark a method that should be executed before each test method in a test class.
    • Common tasks performed in a method annotated with @Before include initializing objects, setting up mock objects, and preparing the test environment.
    • Annotated methods should have a void return type and take no parameters.
  1. @After:
    • Similarly, @After is used to mark a method that should be executed after each test method in a test class.
    • Common tasks performed in a method annotated with @After include releasing resources, closing connections, and resetting objects.
    • It also should have a void return type and take no parameters.

what is the significance of _class field ?

  • When we retrieve documents from MongoDB, Spring Data MongoDB uses the _class field to determine the appropriate Java class to instantiate.
  • For example, when querying for an animal (parent class) document, Spring Data MongoDB checks the value of the _class field and instantiates a Dog (one child class) object if the value is "com.example.animals.Dog", or a Cat (another child class)object if the value is "com.example.animals.Cat".
  • This allows us to retrieve the original Java object with its correct type hierarchy, even though MongoDB itself doesn't have native support for storing Java class types.

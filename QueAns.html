Spring Que - Ans

@Before:
@Before is an annotation used to mark a method that should be executed before each test method in a test class.
The annotated method will run before each test method, ensuring that any necessary setup tasks are performed consistently before each test.
Common tasks performed in a method annotated with @Before include initializing objects, setting up mock objects, and preparing the test environment.
Annotated methods should have a void return type and take no parameters.

What is the process of Spring initialisation ? Complete step by step with demonstration thru logs.

Spring Initialization :

  1. Application Startup: Spring Boot initializes the application context when the application starts.
  1. Profile Check : It checks for the any specific Profile. If not specified it will run as a “default” profile.
  1. Component Scanning: Spring scans the classpath for components, such as beans annotated with @Component, @Service, @Repository, or @Controller.
  1. Bean Registration and nitialization: Spring registers these components as beans in the application context. Also Spring initializes the beans, which involves instantiating them and configuring their properties.
  1. Dependency Injection: Spring injects dependencies into the beans.
  1. Servelet Starts : It starts the server. In our case it starts tomcate server.

What is the difference between Auto-wiring a Service or getting it from ApplicationContext using SpringLocatorFactory?

  1. Auto-wiring a Service:
    • Auto-wiring is a mechanism provided by Spring for automatic dependency injection.
    • When you auto-wire a service, Spring automatically injects an instance of the service into the dependent component (such as a controller, another service, or a bean) based on the specified injection type (e.g., constructor injection, setter injection, or field injection) and the matching bean definition in the application context.
    • Auto-wiring is convenient and commonly used, as it reduces the need for explicit configuration and manual instantiation of dependencies.

  1. Getting it from ApplicationContext using SpringLocatorFactory:

    • Using SpringLocatorFactory, We can typically obtain a reference to the ApplicationContext and then retrieve the desired bean by its name or type.
    • This approach gives more control over the bean retrieval process.
    • However, it may introduce tighter coupling between components and may not follow the recommended dependency injection principles advocated by Spring.

Que : 4

Diff between Auto-wiring using a constructor or separately as a variable ?

Constructor Injection

  • In constructor injection, dependencies are injected through the constructor of a class.
  • Constructor injection is considered a best practice in Spring because it ensures that all required dependencies are provided when an instance of the class is created. This makes the dependencies explicit and immutable once set.
  • Constructor injection promotes better testability, as dependencies can be easily mocked or stubbed during unit testing.
  • Constructor injection also helps in enforcing immutability and promoting thread safety.

Field (Variable) Injection

  • In field injection, dependencies are injected directly into class fields using @Autowired annotations.
  • However, field injection can lead to hidden dependencies and can make the code harder to understand, as dependencies are not explicitly declared in the constructor.
  • Field injection can also make unit testing more challenging, as it may require the use of frameworks like Mockito to mock dependencies.

How to deal with Cyclic dependency ?

  1. @Lazy method :
    • By default, Spring creates all singleton beans eagerly at the startup/bootstrapping of the application context.

    public class A {
    private B b;

    @Autowired
    public A(@Lazy B b) {
        this.b = b;
    }
}

    • when Spring creates the application context, it will create a proxy for the B bean instead of creating the actual bean instance immediately. When the B bean is needed (e.g., when injecting it into A), Spring will initialize the B bean and inject it into A. This breaks the circular dependency because A does not need to be initialized when creating B.

  1. Using @PostConstruct
    • Another way to break the cycle is by injecting a dependency using @Autowired on one of the beans and then using a method annotated with @PostConstruct to set the other dependency.

    Example :

    public class BeanB {

    BeanB() {
        System.out.println("BeanB constructor called !!!");
    }

    private BeanA beanA;
    
    @PostConstruct
    public void init(){
        beanA.setBeanB(this);
    }

    @Autowired
    public void setBeanA(BeanA beanA) {
        System.out.println("Setting property beanA of BeanB instance");
        this.beanA = beanA;
    }
}

And don’t inject dependency using @Autowired in other class. Keep it plain as below.

@Component
public class BeanA {

    BeanA() {
        System.out.println("BeanA constructor called !!!");
    }

    private BeanB beanB;

    public void setBeanB(BeanB beanB) {
        System.out.println("Setting property beanB of BeanA instance");
        this.beanB = beanB;
    }
}

Is getting a cyclic dependency at first place is a precedent for wrong design or there can be a valid use case for this ?

  • When we have a circular dependency, it’s likely we have a design problem and that the responsibilities are not well separated. It is not following the “Single Responsibility Principle”.
  • We should try to redesign the components properly so that their hierarchy is well designed and there is no need for circular dependencies.
  • However, there are many possible reasons we may not be able to do a redesign, such as legacy code, code that has already been tested and cannot be modified, not enough time or resources for a complete redesign, etc.
  • So in nutshell, Having a circular dependencies shows it is a bad design.

what happens if an interface has an implementation and if you try to Autowire that implmentation in the constructor without Qualifier ?

There are Two cases:

  1. Only one implementation of that class is present:
    • In such case it will create a bean of that implementation class automatically.
  1. More than one implementation class is present :
    • In such case it will throw an error saying more than one implementation is present for creating the bean.
    • In such cases we can use Qualifier to specify which implementaion to use.
    • Or we can put @Primary above one of the implementation class to tell that if more than one implementation is present use this class to create a bean.

Hidden Dependencies:

The dependency is hidden within the class as a private field. Without examining the implementation of class, it's not immediately apparent from the outside that it relies on an private field dependency.

what is MOCKITO @Befor and @After ?

  1. @Before:
    • @Before is an annotation used to mark a method that should be executed before each test method in a test class.
    • Common tasks performed in a method annotated with @Before include initializing objects, setting up mock objects, and preparing the test environment.
    • Annotated methods should have a void return type and take no parameters.
  1. @After:
    • Similarly, @After is used to mark a method that should be executed after each test method in a test class.
    • Common tasks performed in a method annotated with @After include releasing resources, closing connections, and resetting objects.
    • It also should have a void return type and take no parameters.
